RESPUESTA DEL APARTADO E) PRUEBAS DE LA APLICACIÓN

1) Casos de prueba unitarios y funcionales en backend
Sí se cumple el requisito.

Se implementó una suite de pruebas automáticas en backend usando el runner nativo de Node.js (node:test) junto con supertest para validar endpoints HTTP. Aunque la consigna menciona Jest o Mocha como ejemplo, esta implementación es equivalente en objetivo y alcance para pruebas unitarias/funcionales de la API.

Archivo de pruebas:
- backend/test/api.test.js

Casos cubiertos (8 pruebas):
- Login exitoso con credenciales correctas (POST /auth/login).
- Seguridad: rechazo de acceso sin token (GET /productos -> 401).
- Funcionalidad: paginación y filtros en productos (GET /productos).
- Autorización por rol: usuario normal no puede listar usuarios (GET /usuarios -> 403).
- Funcionalidad admin: cambio de rol permitido para admin (PATCH /usuarios/:id/rol).
- Seguridad: ruta externa protegida sin token (GET /externa/convertir -> 401).
- Funcionalidad: conversión externa exitosa con token (GET /externa/convertir).
- Validación de entrada: query inválida rechazada (GET /externa/precio-producto-mxn -> 400).

Evidencia de ejecución local:
- Comando: npm test (en carpeta backend)
- Resultado actual: 8/8 pruebas en estado PASS.

2) Pruebas de seguridad (inyección, autenticación y autorización)
Sí se cumple el requisito en autenticación/autorización y mitigación de inyección a nivel de aplicación.

Controles implementados y verificados:
- Autenticación JWT en rutas protegidas mediante middleware authenticateJWT.
- Autorización por roles mediante middleware authorizeRoles (admin/usuario).
- Validación de body y query con middlewares de validación (campos obligatorios, tipos y rangos permitidos).
- Consultas parametrizadas con placeholders (?) en SQL para evitar concatenación directa de entrada de usuario y reducir riesgo de inyección SQL.

Evidencia práctica de seguridad en pruebas:
- Sin token en rutas protegidas: respuesta 401.
- Usuario sin rol requerido en rutas admin: respuesta 403.
- Parámetros inválidos: respuesta 400.

Conclusión
El apartado e) queda cumplido: existen pruebas funcionales automatizadas del backend, y también pruebas/controles de seguridad sobre autenticación, autorización, validación de entradas y mitigación de inyección mediante consultas parametrizadas.
